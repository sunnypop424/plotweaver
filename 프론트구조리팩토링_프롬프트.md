# 플롯위버 웹 — 프론트 구조 리팩토링 프롬프트

> **용도**: `web/` React 앱을 라우팅 기반으로 재구성하고, 레이아웃·공통 컴포넌트를 분리하며,
> 화면 프론트 설계를 최적화하고 파일 구분을 명확히 하기 위한 단일 실행 프롬프트.
> **첨부 권장**: `DESIGN.md`(토큰) + 이 문서. (개별 화면을 다시 디자인하는 게 아니라 **구조만 재편**한다.)
> **대전제**: ★회귀 금지는 구조 리팩(STEP 1–5)에만 적용. **STEP 6(디자인 통일)**은 화면 간 불일치를 수정하는 것이 목적이므로, DESIGN.md 토큰에 맞게 의도적으로 값을 통일한다.

## 0. 컨텍스트 (현재 상태 — 2026-06-29 기준, 그대로 신뢰해도 됨)

- 스택: React 18 + Vite 6 + TypeScript + Tailwind 3. 경로 `web/`.
- **라우터 미설치.** `src/App.tsx`가 `useState`와 좌하단 개발용 스위처로 **22개 화면**을 전환한다.
  - `App.tsx` 주석에 실서비스 라우트 단서 존재: `C1 /create · C2 /create/relations · C3 /create/generating`.
- 화면: `src/screens/*.tsx` **22개** (평면 구조).
- 공통 컴포넌트: `src/components/` — `WizardChrome`, `HybridSelect`, `WorkCard`, `LibraryCard`, `CoverTile`, `TipModal`, `Toast`.
- 훅: `src/lib/useViewport.ts` — `{ vw, isMobile, isDesktop }` 반환.
- 디자인 토큰: `DESIGN.md` 단일 출처 → `tailwind.config.js` / `index.css`의 커스텀 유틸(`pw-btn-*`, `brand`/`ink`/`ink2`/`wash`/`hairline` 등).

### 0-1. ★창작(C) 위저드 = 4스텝이 각각 별도 화면 (이번 리팩토링의 핵심)

| 스텝 | 화면 파일 | `WizardChrome current` |
|---|---|---|
| ① 기본설정 | `C1SettingsWizard` | 1 |
| ② 서사설정 | `CNarrativeWizard` | 2 |
| ③ 관계도 | `C2RelationshipBuilder` | 3 |
| ④ 출력설정 | `COutputWizard` | 4 |
| (생성) | `C3GeneratingLoader` | — |

> 네 스텝 화면이 **모두 `WizardChrome`(상단바+4스텝 인디케이터)를 개별 렌더**하며 `current`/`onBack`/`onSaveDraft`만 다르게 넘긴다. → **WizardLayout으로 통합할 1순위 대상.**

### 0-2. 확인된 중복/안티패턴

1. 위저드 4스텝 + C3가 `WizardChrome`을 **각자 호출**(레이아웃 미부재).
2. 거의 모든 화면이 `sticky top-0` **상단바/헤더를 화면 내부에 인라인 재구현**(위저드형 / 마켓형 `h-[60px]`(M·D·P) / 리더형(블러) / 판매자형 / 어드민형).
3. `Toast`/`useToast`를 **거의 전 화면이 각자 import·렌더**(전역 토스트 부재).
4. `useViewport`를 다수 화면이 개별 호출(레이아웃 1회로 충분한 경우 다수).
5. **크로스-화면 import**: `COutputWizard`가 `C1SettingsWizard`에서 `RadioPill`을 import(`import { RadioPill } from "./C1SettingsWizard"`). 화면이 다른 화면에 의존 → 공통 컴포넌트로 추출해야 함. (`C1`에 `export`로 노출된 다른 헬퍼가 더 있는지도 점검.)
6. 화면 파일이 평면 배치라 **도메인(창작/마켓/서재/결제/판매자/운영) 경계가 파일 트리에 드러나지 않음**.
7. 더미 데이터 배열이 화면마다 하드코딩(WORKS, LIBRARY, TX, SLIDES 등).

## 1. 목표 (Definition of Done)

1. **라우팅 도입**: `react-router-dom` 기반 선언적 라우트. 개발용 스위처는 제거하거나 라우트로 대체.
2. **레이아웃 분리**: 반복되는 상단바/셸을 도메인별 **Layout 컴포넌트**로 추출, 중첩 라우트의 공통 셸로 사용.
3. **공통 컴포넌트 분리**: 화면마다 복붙된 UI + 크로스-화면 import(RadioPill 등)를 재사용 컴포넌트로 승격.
4. **파일 구조 명확화**: 도메인/역할별 디렉터리로 재배치, 경로 별칭(`@/`) 도입.
5. **구조 회귀 0**: STEP 1–5 작업 후 모든 화면이 리팩토링 전과 픽셀·인터랙션 동일. `tsc --noEmit` + `vite build` 통과.

## 2. 작업 순서 (반드시 단계별로, 각 단계 후 빌드 확인)

### STEP 1 — 라우팅 골격
- `react-router-dom` 설치. `createBrowserRouter` + `RouterProvider`(또는 `<Routes>`)로 라우트 정의.

| 인증 | `/auth` | `AAuthOnboarding` | PublicLayout |
| 온보딩 | (오버레이) | `ONBOnboarding` | 전역 오버레이(아래 주1) |

### STEP 2 — 레이아웃 분리
화면 내부에 인라인된 셸을 다음 레이아웃으로 추출하고, 중첩 라우트의 부모(`<Outlet/>`)로 사용한다.

- **`WizardLayout`** *(최우선)* — `WizardChrome`(상단바 "← 뒤로 / 새 작품 만들기 / 임시저장" + 4스텝 인디케이터)을 흡수. C1·CNarrative·C2·COutput 4개 화면이 공유하고, **`current` 단계는 라우트별로 주입**(라우트 메타 또는 `<Outlet context>` 활용). `onBack`/`onSaveDraft` 기본 동작도 레이아웃이 제공하되 기존 토스트 문구 보존.
- `MarketLayout` — 마켓/서재/작품상세/마이/결제 공통 상단 헤더(`sticky top-0 h-[60px]`: 로고·검색·잉크/크레딧·프로필). M·D·P 계열 헤더 단일화.
- `ReaderLayout` — 몰입형 리더 바(반투명/블러, 테마 전환). `MReader` 전용.
- `SellerLayout` — 판매자 콘솔 헤더/탭. S 계열.
- `AdminLayout` — 어드민 상단바. `OAdminConsole`.
- `PublicLayout`(선택) — 랜딩/인증용 최소 래퍼.

원칙:
- 레이아웃이 `useViewport`를 **1회** 호출해 `{ vw, isMobile, isDesktop }`를 Context로 하위에 공급 → 화면 중복 호출 제거(화면은 `useViewportContext()`로 소비).
- 레이아웃은 **시각적으로 기존과 동일**해야 한다(높이·패딩·z-index·border·blur 값 보존).

### STEP 3 — 전역 토스트
- `ToastProvider` + `useToast()` 훅으로 전역화. 루트(또는 각 레이아웃)에서 1회 마운트.
- 전 화면의 개별 `<Toast/>` 렌더와 로컬 `useToast` import 제거, 호출부(`showToast(...)`)만 전역 훅으로 교체.
- 토스트 디자인 토큰(`toast #242537`, 모션 0.2s)·위치 그대로.

### STEP 4 — 공통 컴포넌트 승격
화면 간 반복 UI + 크로스-화면 의존을 `components/`로 추출:
- **`RadioPill`** — `C1SettingsWizard`에서 `export`되어 `COutputWizard`가 import 중. → `components/ui/RadioPill.tsx`로 이동, 두 화면 모두 거기서 import. (C1에서 노출 중인 다른 공유 헬퍼도 동일 처리.)
- `Tabs`/`SegmentedControl` — `PMyPage`·`SDashboard` 등 탭 UI.
- `FilterPills` — `DLibrary`·`MMarketHome`의 필터칩.
- `EmptyState` — 빈 서재/빈 검색 등.
- `MobileCTABar` — `<768`에서 반복되는 하단 고정 CTA 바.
- `StatusBadge`/`GradeBadge` — `DWorkDetail`의 STATUS_MAP/GRADE_MAP, `WorkCard`의 `Badge` 등 표기 요소 정리·재사용.
- 추출 기준: **2곳 이상에서 동일/유사 마크업 반복** 또는 **크로스-화면 import**면 컴포넌트화. 1곳뿐이면 화면에 둔다(과도한 추상화 금지).

### STEP 6 — 디자인 디테일 통일

> **목적**: DESIGN.md 토큰을 유일한 기준으로, 화면 간 불일치를 제거하고 반복 패턴을 공통 컴포넌트로 추출해 전체 UI의 통일성을 높인다. STEP 1–5 이후 진행하거나, STEP 4 공통 컴포넌트 승격과 병행해도 됨.

  components/

- import 경로 전부 갱신, 죽은 import/크로스-화면 import 제거.

## 3. 제약 / 가드레일

- **디자인 토큰은 `DESIGN.md` 단일 출처.** 색·간격·라운드·모션 값을 새로 만들거나 바꾸지 말 것. 기존 Tailwind 토큰/`pw-*` 유틸 그대로 사용.
- **시각/동작 회귀 금지.** 마크업을 옮기되 클래스·구조·z-index·애니메이션은 보존. 의심되면 옮기지 말고 그대로 둔다.
- 한 번에 한 STEP. 각 STEP 후 `npm run typecheck`(= `tsc --noEmit`)와 `npm run build` 통과 확인.
- 외부 의존성 최소화: 라우터(`react-router-dom`)만 추가 허용. UI/상태관리 라이브러리 신규 도입 금지.
- 과잉 추상화 금지: 실제 중복(2곳+) 또는 크로스-화면 import만 컴포넌트화.
- 컴포넌트는 주변 코드 컨벤션(네이밍, 주석 밀도, Tailwind 인라인 스타일)을 따른다.
- 파일 이동은 영역별로 점진적으로. 거대한 단일 변경 지양.
