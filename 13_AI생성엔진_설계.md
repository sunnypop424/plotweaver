> 플롯위버의 심장 — "설정 → 회차 본문 → 표지"가 실제로 어떻게 만들어지는가
> 작성: 수석 PM / 최종수정: 2026-06-29

## 0. 이 문서의 목적과 범위

`11_기능명세서`의 **F-12(회차 본문 생성)·F-13(재생성)·F-14(표지 생성)**, `12_DB_API_명세`의 **generation_jobs / 생성 API**가 "무엇을 한다"를 정의했다면, 본 문서는 그 안에서 **"실제로 어떤 프롬프트를, 어떤 컨텍스트로, 어떤 순서로 LLM에 던져 일관성을 유지하는가"**를 정의한다.

> 🎯 본 문서가 막는 P0 리스크 2종 (`00_1pager`)
> - **품질**: "30화 일관성 붕괴(캐릭터 붕괴)" → 본 문서의 **일관성 엔진(§3·§4)**이 직접 대응.
> - **원가**: "AI 원가 > 판매가" → 본 문서의 **컨텍스트 압축·캐싱 전략(§4·§7)**이 토큰을 줄여 단위원가를 낮춤. (정밀 단가 측정은 **최종 단계로 이연** — 모델 확정 시점, `09` 실험5·§9. 단 토큰 로깅은 첫날부터 적재.)

> ⚠️ 모델/벤더 표기는 설계 시점 기준의 **추상 역할명**(작문모델·임베딩모델·이미지모델)으로 쓴다. 실제 벤더·모델ID는 PoC 단계에서 품질·원가 측정 후 확정하며, 엔진은 **모델 교체가 쉬운 어댑터 구조**(§6)를 전제로 한다.

## 1. 생성 엔진 전체 개요

### 1-1. 입력 → 출력 한눈에

```
[작품 설정(02·03)]   [인물·관계도(F-11)]   [이전 회차 누적 맥락]
        └───────────────┬───────────────────────┘
                        ▼
            ┌───────────────────────┐
            │   프롬프트 컴포저       │  ← 일관성 엔진의 핵심
            │ (Context Assembler)   │
            └───────────┬───────────┘
                        ▼
   ┌────────────────────────────────────────┐
   │  작문 모델 호출 (회차 본문)              │
   │  · 시스템 프롬프트(세계관 헌법)          │
   │  · 동적 컨텍스트(스토리 바이블+요약+관계)│
   │  · 회차 지시(이번 회차 목표/비트)        │
   └───────────┬────────────────────────────┘
               ▼
   [회차 본문 초안] → [후처리·검증(§5)] → chapters 저장
               │
               └─→ [회차 요약·상태 추출] → 스토리 바이블 갱신(§4)
```

### 1-2. 두 개의 생성 트랙

| 트랙 | 산출물 | 모델 역할 | 핵심 난제 |
| --- | --- | --- | --- |
| **본문 트랙** (F-12/13) | 회차별 텍스트 | 작문 LLM | **장기 일관성**(캐릭터·관계·복선) |
| **표지 트랙** (F-14) | 표지 이미지 | 이미지 생성 모델 | **프롬프트 번역**(설정→비주얼) + 등급 안전 |

두 트랙은 독립 잡이지만 같은 `generation_jobs` 큐를 공유한다(`12` 참조).

## 2. 데이터 모델 (생성 엔진 관점) — 12번 스키마 확장 제안

엔진이 동작하려면 `12_DB_API_명세`의 기존 테이블 외에 **"스토리 바이블/상태"를 저장할 구조**가 필요하다. 아래 테이블은 **2026-06-29 `12`번에 정식 반영됨**(`story_bible`·`chapter_summaries`·`character_state`). 본 절은 그 설계 의도를 설명한다.

### 2-1. 기존 테이블 활용
- `novel_settings` — 불변 설정(시대·장르·목표·갈등·결말·시점·등급·문체 등). **시스템 프롬프트의 원천**.
- `characters` — 인물 카드(외형·성격·역할). **묘사 일관성의 원천**.
- `relationships` — 관계 + **변화 타임라인**(예: "5회차: 동료→연인"). **관계 일관성의 핵심**.
- `chapters` — 회차 본문. `gen_cost`·`gen_model` 로깅 컬럼 활용.
- `generation_jobs` — 비동기 잡 상태.

### 2-2. 추가 제안 테이블/컬럼

```
story_bible            # 작품별 누적 상태(회차가 진행되며 갱신되는 '살아있는 설정')
  novel_id   FK
  synopsis_4act        # 4단 구조(발단/전개/위기/절정·결말) — 회차 분배 기준선
  beat_sheet           # 회차→비트 매핑(예: 1화=발단, 12화=관계전환)
  world_rules          # 세계관 규칙 누적(마법체계 등, 03 World-building)
  glossary             # 고유명사 사전(지명·조직·아이템) — 표기 흔들림 방지
  updated_at

chapter_summaries      # 회차별 요약(다음 회차 컨텍스트용 압축본)
  chapter_id FK
  seq                  # 회차 번호
  summary_short        # 2~3문장 압축(롱컨텍스트 절약용)
  state_delta          # 이 회차에서 바뀐 상태(관계/위치/사망/획득 등) JSON
  open_threads         # 회수 안 된 복선/떡밥 목록

character_state        # 회차 시점별 인물 상태 스냅샷(선택적, 고급)
  character_id FK
  as_of_seq            # 몇 회차 기준 상태인지
  status               # 생존/부상/각성 등 변화 가능 속성
```

> 💡 핵심 아이디어: **원본 설정(불변)** 과 **누적 상태(가변)** 를 분리한다. 작문 모델은 매 회차 "불변 설정 + 현재까지의 가변 상태 요약"만 받으므로, 30화가 되어도 전체 본문을 다시 읽지 않고 일관성을 유지한다.

## 3. 프롬프트 아키텍처 (3계층)

작문 모델 호출은 **3개 계층**을 조립해 만든다. 이 계층 분리가 "일관성 + 토큰 효율"의 핵심이다.

### 계층 1 — 시스템 프롬프트 (세계관 헌법, *거의 불변*)
회차마다 바뀌지 않는 작품의 헌법. **프롬프트 캐싱 대상**(§7).

포함 항목:
- 역할 정의: "너는 한국 웹소설 전문 작가다. 아래 설정을 절대 위반하지 않는다."
- 불변 설정(`novel_settings`): 시대·장르·시점(POV)·문체·톤·결말 방향·연령등급/수위.
- 출력 규약: 분량(예: 보통 4천자), 문단 호흡(웹소설 가독형), 회차 끝 클리프행어 여부, 금칙(03 제약·등급).
- 인물 카드 원본(`characters`): 이름·외형·말투·성격 — **묘사가 회차마다 흔들리지 않도록 고정**.

### 계층 2 — 동적 컨텍스트 (스토리 상태, *회차마다 갱신*)
"지금까지 무슨 일이 있었나"를 압축해 전달. **일관성 엔진의 출력물**(§4).

포함 항목:
1. **4단 구조 + 비트시트**: 전체 줄거리 골격과 "이번 회차가 어디쯤인지".
2. **이전 회차 요약**: 직전 N개 회차의 `summary_short`(전체 본문이 아니라 압축본).
3. **관계 현재 상태**: `relationships`의 타임라인을 **현재 회차 시점으로 해석**한 결과
   (예: 12화 생성 시 → "카엘과 리나는 현재 '연인'. 단, 15화에 '원수'로 전환 예정 — 복선 심을 것").
4. **열린 복선(open_threads)**: 아직 회수 안 된 떡밥 → 회수/유지 지시.
5. **고유명사 사전(glossary)**: 표기 통일.

### 계층 3 — 회차 지시 (이번 회차 미션, *매 호출 신규*)
- 이번 회차 번호/목표 비트(예: "위기 단계 진입: 원수의 정체 암시").
- 직전 회차 마지막 장면(클리프행어 이어받기용 원문 일부).
- 사용자 추가 지시(재생성 시 톤 조정·수정 요청, F-13).

```jsonc
// 작문 모델 호출 페이로드 개념 구조
{
  "system":  "<계층1: 세계관 헌법 + 인물카드>",      // 캐싱
  "context": "<계층2: 4단구조 + 직전요약 + 관계현재상태 + 복선>",
  "task":    "<계층3: 이번 회차 목표 + 직전 마지막 장면 + 사용자지시>",
  "constraints": { "length": 4000, "pov": "3인칭 전지적", "rating": "15세", "end_with": "cliffhanger" }
}
```

## 4. 일관성 엔진 (차별화 핵심)

`01_서비스컨셉`이 약속한 **"관계도 기반 일관성 — 30화가 진행돼도 캐릭터가 무너지지 않음"**을 실제로 구현하는 메커니즘. 4개 장치로 구성된다.

### 4-1. 관계 타임라인 해석기 (Relationship Resolver)
- 입력: `relationships`의 변화 타임라인(`02 §5`: "발단=동료 → 위기=배신자").
- 동작: **현재 생성 중인 회차 번호 → 4단 구조상 어느 단계인지 매핑 → 그 시점의 관계 라벨 산출**.
- 출력(계층2에 주입):
  - 현재 관계 상태("지금은 동료").
  - **다가오는 전환과 그 거리**("3회차 뒤 '배신자'로 전환 — 지금부터 균열의 복선을 깔 것").
- 효과: 관계가 **갑자기** 바뀌지 않고 **빌드업**된다. (단순 생성기와의 결정적 차이)

### 4-2. 스토리 바이블 갱신 루프 (Write-back)
매 회차 생성 **직후** 별도 호출로 상태를 추출해 DB에 다시 쓴다.

```
회차 본문 생성 완료
   ▼
[요약·상태 추출 호출]  → summary_short / state_delta / open_threads 산출
   ▼
story_bible · chapter_summaries · character_state 갱신
   ▼
다음 회차는 '갱신된 상태'를 컨텍스트로 받음  (= 일관성 누적)
```

> 이 루프가 없으면 N화가 늘수록 전체 본문을 다시 읽어야 해 **토큰=원가가 폭발**한다. 요약 누적 방식은 일관성과 원가를 동시에 잡는 설계.

### 4-3. 컨텍스트 예산 관리 (Context Budgeting)
회차가 누적돼도 컨텍스트가 무한정 커지지 않도록 **고정 예산** 안에서 우선순위로 채운다.

| 우선순위 | 컨텍스트 | 압축 정책 |
| :---: | --- | --- |
| 1 | 불변 설정 + 인물 카드(계층1) | 항상 전체(캐싱) |
| 2 | 현재 관계 상태 + 임박 전환 | 항상 전체 |
| 3 | 열린 복선 | 항상 전체(보통 소량) |
| 4 | 직전 1~2회차 요약 | 전체 요약 |
| 5 | 더 이전 회차 | **선택적 회상(RAG)** — 관련 회차만 검색 주입 |

- **5번 = 임베딩 검색(RAG)**: 이번 회차와 의미적으로 관련된 과거 회차 요약만 `chapter_summaries`에서 검색해 주입. (예: 20화에서 3화의 떡밥 회수 시 3화 요약 소환)

### 4-4. 일괄 생성 모드의 일관성
`02 §6`의 "일괄 생성" 선택 시: 회차를 **순차 파이프라인**으로 처리(1화 생성→요약→2화 생성→…). 병렬 생성하지 않는다 — 앞 회차 상태가 뒤 회차 입력이기 때문. 잡 1개 안에서 N회 반복하며 `progress`를 회차 단위로 갱신.

## 5. 후처리 · 품질 검증 파이프라인

생성 직후 본문을 그대로 저장하지 않고 검증 단계를 거친다.

| 단계 | 검사 | 실패 시 |
| --- | --- | --- |
| 5-1 분량 검증 | 목표 분량 ±20% 충족? | 부족 시 "이어쓰기" 보강 호출 |
| 5-2 안전 필터 | 등급(02) 초과 표현? 금칙(03)? | `F-40` 생성단계 필터로 차단/재생성 |
| 5-3 일관성 셀프체크 | 인물 이름·관계·설정 위반? (룰 + LLM 셀프크리틱) | 위반 항목 표시 → 자동 1회 재생성 or 사용자에 경고 |
| 5-4 표기 정규화 | glossary 기준 고유명사 통일 | 치환 |
| 5-5 표절 신호 | 기존작과 과도 유사(`F-50` 연계) | 플래그 |

> 5-3 일관성 셀프체크 = "방금 쓴 회차가 인물카드·관계상태와 모순되지 않는지" LLM에 한 번 더 검수시키는 저비용 가드. 비용/품질 trade-off는 PoC에서 ON/OFF A/B로 결정.

## 6. 표지 생성 트랙 (F-14)

### 6-1. 설정 → 이미지 프롬프트 번역
사용자는 그림 프롬프트를 모른다. 엔진이 **설정을 비주얼 프롬프트로 번역**한다.

```
[제목 · 장르 · 분위기 · 표지스타일(03 §7) · 주인공 외형(characters)]
        ▼  (프롬프트 번역 LLM)
[이미지 모델용 프롬프트: 구도/스타일/색감/인물/타이포 지시]
        ▼
[이미지 생성 모델] → 표지 후보 N장 → 사용자 선택 → covers 저장
```

- 입력 활용: 표지 스타일(웹툰풍/유화풍/실사풍 등), 색감, 인물 포함 여부, 제목 폰트 스타일(`03 §7`).
- 제목 텍스트는 이미지 모델이 글자를 깨뜨릴 수 있으므로 **타이포는 후합성(오버레이) 옵션**을 둔다.

### 6-2. 안전·저작권
- 등급(02)·금칙 반영, 실존 인물/특정 IP 스타일 모방 차단.
- 생성 이미지의 권리·라벨링은 `19_AI생성물_저작권_및_콘텐츠정책` 문서를 따른다.

## 7. 원가·성능 최적화 (P0 원가 리스크 대응)

> 정밀 단가 측정은 **최종 단계로 이연**(`09` 실험5). 여기서는 **엔진이 원가를 줄이는 구조적 장치**만 정의하고, 토큰 로깅은 첫날부터 켜 둔다.

| 장치 | 내용 | 효과 |
| --- | --- | --- |
| **프롬프트 캐싱** | 계층1(세계관 헌법+인물카드)은 회차마다 동일 → 캐시 | 반복 입력 토큰 절감 |
| **요약 누적** | 전체 본문 대신 `summary_short` 주입(§4-2) | 컨텍스트 길이 선형↑ 방지 |
| **선택적 RAG** | 먼 과거는 관련분만 검색 주입(§4-3) | 불필요 컨텍스트 제거 |
| **모델 라우팅** | 요약·상태추출·셀프체크 등 보조 작업은 **저가 모델**, 본문만 고품질 모델 | 보조호출 원가↓ |
| **분량 가드** | 과생성 방지(목표 분량 상한) | 출력 토큰 통제 |
| **재시도 정책** | 실패 시 멱등 재시도, 부분 실패 크레딧 롤백(`12`) | 헛돈 방지 |

- 모든 생성 호출은 **토큰·원가·모델·지연을 로깅**(`F-12` AC, `12` 의 `gen_cost`/`gen_model`). → `15_지표체계`의 원가 대시보드로 집계.

## 8. 비동기 처리 흐름 (12번 API와 연결)

`12_DB_API_명세 §2-3`의 잡 흐름을 엔진 관점으로 상세화.

```
POST /novels/{id}/chapters/generate   (mode·from_seq·count)
   ▼  크레딧 잔액 확인 → 부족 시 403 INSUFFICIENT_CREDIT
   ▼  generation_jobs INSERT (status=queued)
   ▼  큐 적재 → 워커 픽업 (status=running)
   ─────────── 워커 내부 ───────────
   1) 컨텍스트 어셈블(§3·§4): 설정+관계해석+요약+복선 조립
   2) 작문 모델 호출 → 본문 초안
   3) 후처리·검증(§5)  ── 실패 → 재시도 or status=failed(크레딧 미차감)
   4) chapters 저장(gen_cost·gen_model 기록)
   5) 스토리 바이블 갱신 루프(§4-2): 요약·상태 write-back
   6) 성공 시 크레딧 차감 커밋
   ─────────────────────────────────
   ▼  progress 갱신(회차/단계 단위)
GET /jobs/{jobId} 폴링 → 완료 시 result:{chapter_id, seq}
```

- **트랜잭션 경계**: "크레딧 차감"은 본문 저장 성공 후 커밋. 실패 시 전체 롤백 → 사용자 손해 0 (`12` 무결성 규칙).
- **멱등성**: 같은 job 재시도가 회차를 중복 생성하지 않도록 job_id 기준 멱등 처리.
- **대기 UX**: `04 화면 C3(생성 로딩)`의 진행률·예상시간은 워커 progress를 반영.

## 9. PoC 검증 항목 (09 베타실험과 연결)

엔진 설계의 가설은 PoC에서 검증한다. **검증 시점은 두 단계로 분리**한다(검증 순서 결정 2026-06-29, `09` §1):
- **【조기 — 본개발 잠금 해제 전】**: 품질·일관성(핵심 Aha의 전제).
- **【최종 — 개발 후반/출시 직전, 모델 확정 시】**: **토큰 원가 측정 + 요약 누적 방식 검증**(가격은 config로 마지막에 숫자만 조정하는 전략).

| # | 검증 질문 | 측정 방법 | 합격 기준(예시) | **시점** |
| :---: | --- | --- | --- | :---: |
| 1 | 30화까지 캐릭터/관계가 안 무너지나? | 30화 생성 후 일관성 위반 카운트 | 위반 ≤ N건/작품 | 🟢 조기 |
| 4 | 관계 전환이 '빌드업'되어 자연스러운가? | 전환 회차 전후 정성평가 | 독자 평가 통과 | 🟢 조기 |
| 5 | 셀프체크(§5-3)가 비용 대비 효과 있나? | ON/OFF A/B | 효과>비용일 때 채택 | 🟡 중간 |
| 2 | 요약 누적 방식이 전체 본문 주입 대비 품질 유지되나? | 두 방식 블라인드 비교 | 품질 동등 ± | 🔴 **최종** |
| 3 | 회차당 토큰·원가가 분량에 비례 통제되나? | 회차별 토큰 로깅·실원가 측정 | 회차 증가에도 원가 평탄·마진≥0 | 🔴 **최종** |

> ⚠️ #2·#3을 최종으로 미루는 대신, **토큰 로깅(`gen_cost`·`gen_model`)은 본문 생성 첫날부터 켜 둔다**(소급 불가). 최종 검증은 "처음 보는 숫자"가 아니라 "쌓인 로그의 정밀 판정"이어야 한다. 분량·생성단위까지 바꿔야 할 원가면 **구조 변경 리스크**(§10·`16` R1)임을 감수한다.

> 정량 품질 평가 루브릭(캐릭터 일관성·서사 응집·문장 자연성·몰입)은 PoC 시 별도 평가표로 운영하며, 결과를 `15_지표체계`의 품질 가드레일 지표로 승격한다.
